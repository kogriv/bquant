## 0. TL;DR — что такое codemap

Отдельный **статический анализатор**: ест исходники Python-пакета, строит **queryable граф кода** —
фактическую карту, которую потребляют генерация доки, AI/CLI-навигация (RAG) и аудит/чистка.
Философия — как у «доки-как-продукт»: **один канонический граф-стор → много рендеров**.
Не импортирует цель (только `ast`), поэтому работает над любым пакетом и легко выносится в свой репо.

## 1. Зачем и для кого — каталог запросов (краеугольный камень)

Codemap существует ради потребителей. Их конкретные запросы = требования, и они же держат продукт
**конечным** (см. §7). Три потребителя v1:

### Потребитель B — Дока-как-продукт
- Какова **публичная API-поверхность**? (что экспортируется, сигнатуры, докстринги)
  *Нюанс bquant:* поверхность слоёная — `bquant/__init__.py:__all__` отдаёт version+config, а флагман
  `analyze_zones` живёт в `bquant.analysis.zones`. Граф должен различать «top-level export» и «глубокий».
- Что принадлежит модулю X (классы/функции/dataclass'ы)?
- Что помечено `@deprecated` (напр. `MACDZoneAnalyzer`)?
- Дай докстринг+сигнатуру символа Y для reference-страницы.

### Потребитель A — CLI-AI тул / RAG / навигация
- Где реализована фича X? (символ → файл:строки)
- Что зависит от Y / от чего зависит Y? (рёбра импортов)
- Дай **«минимал»** для символа Z: сигнатура + докстринг + место + ближайшие соседи — чанк для RAG.

### Потребитель C — Аудит / гигиена
- Мёртвое/недостижимое: модули без входящих ссылок, символы без ссылок.
- Затенённые/дублирующие имена; циклы зависимостей.
- Public-but-undocumented; экспорт без докстринга.
  *(Прим.: прежние кандидаты `base_old.py`/`library.py`-тень уже вычищены — capability остаётся.)*

> **Это и есть спека v1:** реализуем ровно этот каталог запросов, не больше. Новый вопрос →
> осознанное расширение, а не «на всякий случай».

## 2. Модель графа (узлы / рёбра / атрибуты)

Выводится из §1 — минимальный набор, чтобы ответить на запросы.

**Узлы (kind):** `Package`, `Module`, `Class` (dataclass — подтип с флагом), `Function`/`Method`,
опционально `Constant`. Параметры — атрибут callable-узла, не отдельные узлы (v1).

**Рёбра:**
- `contains` — модуль → класс/функция (структура).
- `imports` — модуль → модуль/символ (зависимости).
- `inherits` — класс → базовый класс.
- `exports` — модуль → символ (через `__all__`/публичность).
- `decorated_by` — символ → декоратор (ловит `@deprecated`, `@StrategyRegistry.register_*`).
- `references`/`calls` — **best-effort статически; точный граф вызовов отложен** (см. §7).

**Атрибуты узла:** qualified name, kind, `file:line-range`, signature (для callable), docstring,
visibility (public/private по `_`/`__all__`), список декораторов, `is_deprecated`, `is_reexported`.

> Граница конечности: перечисленные рёбра/атрибуты статически детерминированы → **конечны**.
> Точная трассировка потоков/вызовов через рантайм — слой глубины, не v1.

## 3. Вход — что и как разбираем

- **Статический `ast`-разбор исходников.** Никакого импорта/исполнения цели (безопасность,
  развязка, работает над любым пакетом — в т.ч. неустановленным/сломанным).
- **Цель:** путь к корню пакета. v1 — один Python-пакет (bquant подопытный).
- **Область:** `.py`. Позже (по запросу доки): докстринги-как-данные — уже нужны потребителю B;
  `pyproject.toml` — метаданные пакета.
- **Устойчивость:** файл с синтакс-ошибкой — не валим весь прогон, репортим и пропускаем.

## 4. Что выдаёт — канонический стор + рендеры

**Канон:** один граф-артефакт. Предложение — **JSON** как источник правды (портируемо, диффается,
греп-friendly, версионируется); при необходимости грузим в `networkx`/`sqlite` для запросов в памяти.
JSON канон — ради переносимости и чистого выноса продукта.

**Рендеры поверх канона (все читают один и тот же граф):**
- **Query-API** (библиотека + CLI) — отвечает на запросы §1.
- **Markdown-отчёты** — API-поверхность, мёртвый код, карта зависимостей.
- **Диаграммы** — mermaid/graphviz: граф зависимостей модулей, иерархия классов.
- **RAG-экспорт** — «минималы» по символам как чанки.

> Ровно зеркало «доки-как-продукт»: единый источник, много видов. Граф — единый источник **знания о коде**.

## 5. Архитектура тула (внутри)

Конвейер: **Extract → Build → Store → Serve**
- **Extract** — `ast`-обходчики, по одному на анализ (структура, импорты, наследование, публичность…).
- **Build** — сборка узлов/рёбер в модель.
- **Store** — сериализация в канонический JSON.
- **Serve** — запрос/отчёт/экспорт поверх стора.

**Модульность (это «верстак для экспериментов»):** экстракторы — **плагины**. Новый анализ =
добавить экстрактор, не трогая ядро модели. Ядро маленькое, анализы аддитивны.

## 6. Интерфейс

- **CLI:** `codemap build <path>` → `graph.json`; `codemap query <...>`; `codemap report <kind>`;
  `codemap export-rag <...>`.
- **Библиотечный API** — для программного/AI-вызова (тот же слой Serve).

## 7. Границы v1 (чего НЕ делаем сразу)

- ❌ Импорт/исполнение цели (только статика).
- ❌ Точный динамический call-graph / вывод типов (только best-effort статически).
- ❌ Разрешение динамических импортов, метакласс-магии, monkeypatch (репортим/пропускаем).
- ❌ Кросс-язык, кросс-репо.
- ❌ Веб-UI для людей (это P3 human-view, позже).

## 8. Первый тонкий срез (веха M0)

Один потребитель + один запрос сквозь весь конвейер — проверить форму на практике.
**Предложение M0:** «публичная API-поверхность» для bquant: Extract → Build → `graph.json` →
markdown-отчёт API-surface. Самое маленькое, что доказывает конвейер end-to-end и **сразу полезно**
(вход для потребителя B). Дальше расширяем каталог §1.

## 9. Структура каталога `codemap/` (предложение)

```
codemap/
├── README.md          # что это, статус, прицел на вынос
├── DESIGN.md          # этот документ
├── codemap/           # код тула (extract/build/store/serve)
│   ├── extract/       # ast-экстракторы (плагины)
│   ├── model.py       # узлы/рёбра/атрибуты
│   ├── store.py       # JSON канон
│   └── serve/         # query / report / export
├── tests/
└── (позже) pyproject.toml  # при выносе в отдельный репо-продукт
```
BQuant для codemap — **вход**, не зависимость: `codemap build ../bquant`.
