## 0. TL;DR — что такое codemap

Отдельный **статический анализатор**: ест исходники Python-пакета, строит **queryable граф кода** —
фактическую карту, которую потребляют генерация доки, AI/CLI-навигация (RAG) и аудит/чистка.
Философия — как у «доки-как-продукт»: **один канонический граф-стор → много рендеров**.
Статический разбор **без импорта цели**, поэтому работает над любым пакетом и легко выносится в свой репо.

**Стек v1 (решено §10.8):** Python + `griffe` за нейтральным адаптером; JSON-канон; CLI-AI-first
(JSON по умолчанию, детерминизм, `uv`/`pipx`). Мультиязычность/бинарь — позже, добавлением экстракторов.

## 1. Зачем и для кого — каталог запросов (краеугольный камень)

Codemap существует ради потребителей. Их конкретные запросы = требования, и они же держат продукт
**конечным** (см. §7). Три потребителя v1:

### Потребитель B — Дока-как-продукт
- Какова **публичная API-поверхность**? (что экспортируется, сигнатуры, докстринги)
  *Нюанс bquant:* поверхность слоёная — `bquant/__init__.py:__all__` отдаёт version+config, а флагман
  `analyze_zones` живёт в `bquant.analysis.zones`. Граф должен различать «top-level export» и «глубокий».
- Что принадлежит модулю X (классы/функции/dataclass'ы)?
- Что помечено `@deprecated` (напр. `MACDZoneAnalyzer`)?
- Дай докстринг+сигнатуру символа Y для reference-страницы.

### Потребитель A — CLI-AI тул / RAG / навигация
- Где реализована фича X? (символ → файл:строки)
- Что зависит от Y / от чего зависит Y? (рёбра импортов)
- Дай **«минимал»** для символа Z: сигнатура + докстринг + место + ближайшие соседи — чанк для RAG.

### Потребитель C — Аудит / гигиена
- Мёртвое/недостижимое: модули без входящих ссылок, символы без ссылок.
- Затенённые/дублирующие имена; циклы зависимостей.
- Public-but-undocumented; экспорт без докстринга.
  *(Прим.: прежние кандидаты `base_old.py`/`library.py`-тень уже вычищены — capability остаётся.)*

> **Это и есть спека v1 (зафиксировано):** реализуем ровно этот каталог запросов, не больше.
> Новый вопрос → осознанное расширение, а не «на всякий случай».
> **M0 (§8) берёт один срез — публичную API-поверхность (потребитель B);** остальные запросы —
> по мере расширения после M0.

## 2. Модель графа (узлы / рёбра / атрибуты)

> **Как называем сущность:** **код-граф** (артефакт, что строит codemap). С точки зрения
> свежести — производный **индекс/кэш кода**, что и диктует стратегию его обновления (§15).
> Шире — это **граф знаний о системе**, где код — домен №1 (§16).

> **Kind — открытые (строка + атрибуты), не закрытый enum.** Иначе будущие домены (сервис,
> сервер, деплой — §16) требовали бы хирургии модели. Ядро знает «узел = kind + attrs{}»,
> экстрактор кладёт специфику (Python — `is_dataclass`; инфра — `image`, `replicas`).

Выводится из §1 — минимальный набор, чтобы ответить на запросы.

**Узлы (kind):** `Package`, `Module`, `Class` (dataclass — подтип с флагом), `Function`/`Method`,
опционально `Constant`. Параметры — атрибут callable-узла, не отдельные узлы (v1).

> Модель — **язык-нейтральное ядро** (модуль/символ/зависимость/публичность есть почти везде).
> Python-измы (`dataclass`, декораторы) — не в ядре, а **extra-атрибуты от экстрактора**. См. §12.

**Рёбра:**
- `contains` — модуль → класс/функция (структура).
- `imports` — модуль → модуль/символ (зависимости).
- `inherits` — класс → базовый класс.
- `exports` — модуль → символ (через `__all__`/публичность).
- `decorated_by` — символ → декоратор (ловит `@deprecated`, `@StrategyRegistry.register_*`).
- `references`/`calls` — **best-effort статически; точный граф вызовов отложен** (см. §7).

**Атрибуты узла:** qualified name, kind, `file:line-range`, signature (для callable), docstring,
visibility (public/private по `_`/`__all__`), список декораторов, `is_deprecated`, `is_reexported`.

**Требование от видов (§4.2):** граф должен уметь отдавать **скоупленные подграфы** — по подсистеме,
публичности, N-соседей вокруг узла. Иначе человеко-читаемые виды нечитаемы на 63 модулях. Это диктует
модель: узлы адресуемы по qualname, рёбра обходимы в обе стороны (нужны и «зависит от», и «от чего зависят»).

> Граница конечности: перечисленные рёбра/атрибуты статически детерминированы → **конечны**.
> Точная трассировка потоков/вызовов через рантайм — слой глубины, не v1.

### 2.1. Идентичность узла и ре-экспорты

**Канонический id узла = полный путь к месту ОПРЕДЕЛЕНИЯ:** `bquant.analysis.zones.pipeline.analyze_zones`.
Один узел на одно определение. Методы — `Class.method`, вложенные классы — через точку.

**Ре-экспорты — это рёбра, а не дубли-узлы.** bquant сразу ломает наивную схему: `analyze_zones`
*определён* в `...pipeline`, а *доступен* как `bquant.analysis.zones.analyze_zones` (через `__init__.py`).
Решение: `export`/`alias`-ребро от точки доступа к каноническому узлу. Тогда оба запроса отвечаемы:
«где реально определён X» (узел) и «как X достижим» (обход alias-рёбер).

**Публичная API-поверхность** (запрос §1) = объединение достижимых экспортов, но каждый резолвится
в одно каноническое определение. Слоёность bquant (`__init__.py:__all__` ≠ глубокие символы) выражается
именно alias-рёбрами, а не размножением узлов.

### 2.2. Набросок JSON-схемы (+ версия)

```json
{
  "codemap_schema": "0.1",
  "target": {"name": "bquant", "root": "bquant"},
  "nodes": [
    {"id": "bquant.analysis.zones.pipeline.analyze_zones", "kind": "function",
     "file": "bquant/analysis/zones/pipeline.py", "lines": [718, 740],
     "signature": "analyze_zones(df: pd.DataFrame) -> ZoneAnalysisBuilder",
     "docstring": "...", "visibility": "public", "decorators": [], "is_deprecated": false},
    {"id": "bquant.indicators.macd.MACDZoneAnalyzer", "kind": "class",
     "decorators": ["deprecated"], "is_deprecated": true, "...": "..."}
  ],
  "edges": [
    {"type": "contains", "from": "bquant.analysis.zones.pipeline", "to": "bquant.analysis.zones.pipeline.analyze_zones"},
    {"type": "export",   "from": "bquant.analysis.zones", "to": "bquant.analysis.zones.pipeline.analyze_zones", "as": "analyze_zones"},
    {"type": "imports",  "from": "bquant.analysis.zones.strategies.swing.zigzag", "to": "bquant.indicators.LibraryManager"}
  ]
}
```

- **`codemap_schema`** — версия формата; бампается при изменении схемы (инвариант, как
  `CACHE_SCHEMA_VERSION` в bquant). Раз граф кормит потребителей и поедет в отдельный репо — версия обязательна.
- **Детерминизм:** содержимое графа сортируется (узлы/рёбра по id) и **не содержит волатильных полей**
  (timestamp и т.п.) → JSON диффается и воспроизводим. Волатильные метаданные (когда собран) — вне
  графа (сайдкар/опционально). *(Урок из практики: встроенные таймстампы ломают воспроизводимость.)*

## 3. Вход — что и как разбираем

- **Статический `ast`-разбор исходников.** Никакого импорта/исполнения цели (безопасность,
  развязка, работает над любым пакетом — в т.ч. неустановленным/сломанным).
- **Цель:** путь к корню пакета. v1 — один Python-пакет (bquant подопытный).
- **Область:** `.py`. Позже (по запросу доки): докстринги-как-данные — уже нужны потребителю B;
  `pyproject.toml` — метаданные пакета.
- **Устойчивость:** файл с синтакс-ошибкой — не валим весь прогон, репортим и пропускаем.

### 3.1. Разрешение импортов (самая хрупкая часть)

bquant — стресс-тест: **глубокие относительные импорты** (реально `from .....indicators import
LibraryManager` — 5 точек), `__init__.py`-ре-экспорты, условные импорты.

> **v1 (решено §10.8): этот резолв выполняет `griffe`** — наш Python-экстрактор его потребляет, не
> реализует. Правила ниже — это **контракт** (что должно быть разрешено), который griffe удовлетворяет
> для Python; для будущих не-Python экстракторов (tree-sitter и т.п.) их пришлось бы реализовать самим.

Стратегия резолва (контракт) — **два прохода**:

1. **Проход 1 — таблица символов:** собрать все определения и экспорты всех модулей.
2. **Проход 2 — резолв рёбер:** каждое `import` резолвить против таблицы.

Правила резолва:
- **Относительные:** считаем точки, идём вверх по дереву пакетов от текущего модуля → абсолютный qualname
  (`.....indicators` из `...swing.zigzag` → `bquant.indicators`).
- **Через `__init__.py`:** цель импорта может оказаться ре-экспортом → доводим по alias-рёбрам (§2.1) до канона.
- **`import *`:** разворачиваем по `__all__` цели, иначе — все публичные имена; неразрешимое — wildcard-ребро + флаг.
- **Условные (`try: import talib`)** — ребро с флагом `conditional`, best-effort.
- **Внешние (`pandas`, `numpy`, …) и неразрешимое** — узел-лист `external`/`unresolved`, **не разворачиваем и не падаем**.

## 4. Что выдаёт — канонический стор + виды

**Канон vs индекс запросов — это РАЗНЫЕ вещи, и в этом ответ на «а почему не БД».**

- **Канонический модель — store-agnostic.** Граф (узлы/рёбра/атрибуты) определён независимо от
  хранилища. Сериализация по умолчанию — **JSON** (портируемо, диффается, греп-friendly, версионируется,
  без сервера — ради выноса продукта и хартии §14.1). Тот же граф тривиально ложится в SQLite-таблицы
  или узлы/связи Neo4j.
- **Query-слой — адаптер** (как §13): интерфейс запросов + сменные бэкенды. Виды/RAG/CLI зовут
  **интерфейс**, не хранилище. Апгрейд по масштабу, **не переписывание**:

| Бэкенд запросов | Когда | Порог |
|---|---|---|
| **in-memory (`networkx`)** из JSON | v1, малый граф (bquant — сотни узлов) | по умолчанию |
| **SQLite** (производный индекс из JSON) | тёплый serve / бóльший граф / индексированные запросы | масштаб/производительность |
| **Neo4j** | **на порядок сложнее** — граф микросервисов+инфры, тяжёлый traversal (§16) | system-scale, приняв сервер |

> Проекты на Neo4j/SQLite обычно на большом масштабе (org-wide, миллионы файлов) — другой режим.
> Neo4j — required-сервер, поэтому вне system-scale он против хартии §14.1; door к нему держим открытой
> именно через store-agnostic канон + сменный query-бэкенд, а не встраивая его в ядро.

**Всё остальное — виды (renderers) поверх одного канона.** Делятся на машинные и человеко-читаемые.

**Машинные виды:**
- **Query-API** (библиотека + CLI) — отвечает на запросы §1.
- **RAG-экспорт** — «минималы» по символам как чанки.

### 4.1. Каталог человеко-читаемых видов

Человеку граф показывают четырьмя «жанрами». Порядок — от дешёвого/полезного к мощному/тяжёлому.

**D. Структурные отчёты (markdown, не диаграммы) — максимум пользы на строчку**
- API-surface (дерево публичных символов + сигнатуры/докстринги); карта зависимостей (кто кого
  импортирует, циклы); мёртвый код; матрица зависимостей модуль×модуль; «public vs documented».
- Текст, версионируется, читают и человек, и ИИ, кормят доку напрямую. **Часть M0.**

**B. Obsidian-стиль (связанный markdown-vault)**
- По заметке на символ/модуль, сшитые `[[wikilinks]]` (импорты / наследование / «определён в»).
- Obsidian бесплатно даёт: интерактивный граф, backlinks, локальные графы вокруг узла, поиск.
- **Не строим свой граф-движок — переиспользуем чужой.** Те же заметки работают и как обычная дока.

**A. Текстовые диаграммы (встраиваются в markdown)**
- **Mermaid** — `classDiagram` (наследование), dep-граф модулей; рендерится в GitHub/Obsidian,
  дёшево, плохо тянет большие графы.
- **PlantUML** — полноценный UML (class / package / component); богаче, но нужен свой рендерер.
- **Graphviz/DOT** — большие dep-графы с автолейаутом; статичная картинка.

**C. Интерактивный веб-граф**
- D3 / Cytoscape / pyvis — зумируемый кликабельный граф с поиском. Мощно для навигации по большому
  пакету, но тяжелее (свой веб-слой, хостинг). Это уже «настоящий human-view P3».

### 4.2. Оговорка масштаба (требование к графу)

У bquant 63 модуля и сотни символов. **Одна мега-диаграмма всего пакета нечитаема.** Любой вид обязан
**скоупиться**: per-module, per-subsystem, отфильтрованный (только публичное / только зоны /
N-соседей вокруг символа). → прямое требование к §2: граф должен отдавать **скоупленные подграфы**.

### 4.3. Рекомендуемый порядок (дёшево → мощно)

1. **Структурные markdown-отчёты (D)** — часть M0, польза сразу, кормят доку.
2. **Obsidian-vault (B)** — «граф-ощущение» почти бесплатно, переиспользуя Obsidian.
3. **Скоупленные mermaid-диаграммы (A)** — встроить в отчёты (class-диаграмма подсистемы, dep-граф).
4. **Интерактивный веб-граф (C)** — позже, когда упрёшься в статику (полноценный P3).

**Отложено:** sequence / call-flow диаграммы — им нужен точный call-graph (слой глубины §7), не v1.

> Ровно зеркало «доки-как-продукт»: единый источник, много видов. Граф — единый источник **знания о коде**.

## 5. Архитектура тула (внутри)

Конвейер: **Extract → Build → Store → Serve**
- **Extract** — `ast`-обходчики, по одному на анализ (структура, импорты, наследование, публичность…).
- **Build** — сборка узлов/рёбер в модель.
- **Store** — сериализация в канонический JSON.
- **Serve** — запрос/отчёт/экспорт поверх стора.

**Модульность (это «верстак для экспериментов»):** экстракторы — **плагины**. Новый анализ =
добавить экстрактор, не трогая ядро модели. Ядро маленькое, анализы аддитивны.

**Границы стадий — это контракты (данные), а не вызовы.** Extract и Serve общаются с ядром через
нейтральную схему (§2.2), поэтому любую стадию можно заменить внешним инструментом за адаптером — см. §13.

## 6. Интерфейс

- **CLI:** `codemap build <path>` → `graph.json`; `codemap query <...>`; `codemap report <kind>`;
  `codemap export-rag <...>`. **JSON по умолчанию** + человеческие форматы; стабильные exit-коды,
  структурные ошибки, `--dry-run` для мутаций (конвенции AI-CLI-first, §14.1).
- **Библиотечный API** — для программного/AI-вызова (тот же слой Serve).
- **(Позже, опционально) тёплый serve-режим** — MCP-сервер / локальный демон с графом в памяти для
  горячего AI-пути (ноль startup на вызов). Без уникальной логики — те же сервисы Serve. См. §14.4.

## 7. Границы v1 (чего НЕ делаем сразу)

- ❌ Импорт/исполнение цели (только статика).
- ❌ Точный динамический call-graph / вывод типов (только best-effort статически) →
  поэтому sequence / call-flow диаграммы тоже вне v1 (см. §4.3).
- ❌ Разрешение динамических импортов, метакласс-магии, monkeypatch (репортим/пропускаем).
- ❌ Кросс-язык, кросс-репо.
- ❌ Веб-UI для людей (это P3 human-view, позже).

**Отдельно — динамическая проводка (registry/factory).** bquant связывает поведение не импортами,
а декораторами и фабриками: `@StrategyRegistry.register_swing_strategy('zigzag')`,
`IndicatorFactory.create('pandas_ta', 'zigzag')`. Статически граф видит **декоратор** и, если аргумент —
строковый литерал, **пишет его в атрибут** (так «какая стратегия под каким ключом» частично
восстановимо). Но сам рантайм-мэппинг `'zigzag' → ZigZagSwingStrategy` и диспетч фабрики
**не резолвятся** в v1. Такие места помечаем флагом `dynamic-registration`, чтобы потребитель знал:
здесь граф **структурен, а не поведенчески полон** — иначе он вводит в заблуждение.

## 8. Первый тонкий срез (веха M0)

Один потребитель + один запрос сквозь весь конвейер — проверить форму на практике.

## 11. Prior art и build-vs-buy

Не в вакууме — что уже существует и что из этого подсматриваем/берём:

| Инструмент | Что делает | Отношение к codemap |
|---|---|---|
| **`griffe`** (mkdocstrings) | статик-разбор Python → структурная API-модель (модули/классы/функции, сигнатуры, докстринги, часть резолва импортов/алиасов) | **ближайший prior art** — делает половину §2/§3, включая самую хрупкую часть (§3.1) |
| **`pydeps`** | граф импортов/зависимостей (модульный уровень) | идеи для dep-видов (§4) |
| **`pyan3`** | статический call-graph | референс для отложенного слоя вызовов (§7) |
| **`import-linter`** | контракты/слоистость импортов | идея для аудит-запросов (§1-C) |
| **`pdoc` / `sphinx.apidoc`** | генерация API-доки из интроспекции | что НЕ переизобретаем в потребителе B |

**Нативные анализаторы на каждый язык (аналоги griffe) — экстрактор берёт лучший из них:**

| Язык | Нативный анализатор |
|---|---|
| Python | `griffe` (или `ast`) |
| Go | `go/ast` + `go/packages` + `go/types` (stdlib) |
| Rust | `rustdoc --output-format json`, rust-analyzer крейты, `syn` |
| C/C++ | **libclang** / Clang AST |
| JS/TS | **TypeScript compiler API** / `ts-morph` |

`tree-sitter` — единый **фолбэк** для языков без хорошего нативного тула (мельче семантика).

**Позиция:** «свой тулинг» (хартия/роадмап) = владение **продуктом** — граф-моделью, видами, RAG,
верстаком экспериментов и правом на вынос в отдельный репо. Это **не** обязывает переписывать парсер.
Резолв импортов/алиасов — ровно то, что griffe уже делает хорошо и что у нас самое рисковое (§3.1).

**Честный лин:** рассмотреть **griffe как parse/resolve-бэкенд**, а свой код держать поверх (граф-модель
§2 → JSON-канон §2.2 → виды §4 → RAG). «Своё» ≠ «изобрести заново разбор Python». Финальный выбор —
решение §10.8 (griffe-бэкенд vs свой `ast`), и это **самая важная развилка дизайна**: она определяет §3.

## 12. Язык и мультиязычность (архитектурный шов)

Как §3 написан, дизайн Python-специфичен (`ast`, импорты). Но раз codemap метит в отдельный
продукт — правильная форма — **язык-нейтральное ядро + язык-специфичные экстракторы**:

- **Нейтральный спайн** (модель §2, стор §2.2, виды §4, RAG) — общий для всех языков. Понятия
  «модуль/namespace», «символ-определение», «зависимость», «публичность», «наследование» есть почти везде.
- **Язык-специфичный слой — только экстрактор** (§5): парсит исходники одного языка → эмитит в нейтральную
  модель. Python-экстрактор первым (v1). JS/Go/Rust/… — будущие экстракторы, **ядро не трогаем**.
- **Что специфично Python** (`dataclass`, декораторы) — не в ядре, а **extra-атрибуты от экстрактора**.
  Ядро знает «Class + extras{}»; Python-экстрактор кладёт `is_dataclass`, `decorators`.

**Смешанное содержание** (Python + JS + конфиги + SQL + доки): диспетчеризация по типу файла → нужный
экстрактор; неизвестное — **opaque-узел** (файл есть, содержимое не разобрано) + флаг, **не падаем**.

**Trade-off (честно):** мультиязычность **ослабляет** кейс griffe — он Python-only. Чем важнее другие
языки, тем привлекательнее **tree-sitter** (единый парс десятков языков), но ценой потери «бесплатной»
Python-семантики griffe (кросс-файловый резолв импортов §3.1 пришлось бы строить самим per-language).
Это твоё решение и оно двигает §10.8.

**Позиция для v1:** **проектируем нейтрально, реализуем Python-first.** Один Python-экстрактор в v1,
но шов спроектирован так, что добавить tree-sitter-экстрактор позже — **не переделка ядра**.

**Рабочий пример — добавить Go (разобрать `replan`):** ядро/схема/виды/RAG/CLI переиспользуются как
есть; пишешь **Go-экстрактор** — крошечный Go-хелпер на `go/packages`+`go/types`, выдающий нейтральный
JSON, вызываемый **сабпроцессом** (§13), зарегистрированный на `.go`. Работа = обернуть `go/packages`,
ядро не трогаешь. Это и есть момент, ради которого §12/§13 делались: новый язык = **плюс-один
экстрактор**, а не переписывание.

## 13. Интеграция внешних инструментов (адаптеры и открытость)

OSS-baseline — не самоцель, а **референс-реализация, которая ЗАДАЁТ контракты**. Внешние тулы (разной
открытости и тулинга) встраиваются позже — тем же швом, что и языковые экстракторы (§12), обобщённым
до «любой внешний тул за адаптером».

**Контракт = нейтральная граф-схема (§2.2).** Всё внешнее подключается на двух границах:
- **Extract:** исходники → `[адаптер]` → фрагменты нейтрального графа (внешние парсеры/анализаторы).
- **Serve/View:** нейтральный граф → `[адаптер]` → внешние рендеры/RAG/визуализаторы.

Между ними — канонический JSON как lingua franca; ядро не знает, кто по ту сторону адаптера.

**Три режима интеграции (по возрастанию «чёрноящичности») — все за одним интерфейсом адаптера:**

| Режим | Когда | Пример |
|---|---|---|
| **Библиотека (embed)** | тул открыт и импортируется | `griffe` как Python-lib |
| **Subprocess/CLI** | закрытый бинарь / другой язык | tree-sitter, Rust-тул |
| **Сервис/API** | удалённый/проприетарный | облачный анализатор |

**Открытость → максимальная глубина интеграции, не более:** полностью открытый форкаешь/встраиваешь;
открытый API — как зависимость; закрытый — только чёрный ящик через subprocess/API, получаешь ровно что
он отдаёт. Адаптер делает **даже закрытый тул пригодным** — ценой поверхностности.

**Гигиена лицензий/выноса:** codemap метит в отдельный продукт → внешние тулы держим **за адаптерами И
как опциональные extras**, не в ядре. Тогда: (1) shippable-ядро остаётся чисто-лицензионным; (2) закрытый/
тяжёлый тул — opt-in плагин; (3) исчезни внешний тул — baseline всё равно работает.

**План работы с внешними:**
1. **Сначала baseline** (своё/OSS) — фиксирует контракты Extract/Serve, даёт рабочий end-to-end.
2. **Внешние — опциональные апгрейды за адаптером**, A/B против baseline на своём шве (качество/покрытие/
   цена/лицензия). Никогда — жёсткая зависимость.
3. **Реестр кандидатов** — табличка «тул — режим — статус (смотрим/встроен/отклонён) — заметки»; ведём по
   мере оценки. (Первые кандидаты появятся, когда дойдут руки смотреть твои «примеченные».)
4. **Инвариант:** ядро не зависит от закрытого тула; внешнее всегда заменяемо/отключаемо.

## 14. Стек и рантайм (CLI-AI-first)

За образец взят `replan` (Go, headless-first, AI-CLI-first). Его рантайм-принципы переносим,
но со стоп-краном: у codemap выбор стека **не независим** — он сцеплен с §10.8/§12.

### 14.1. Рантайм-хартия (принимаем из replan — стек-независимо)
- **Headless-first, CLI-first MVP.** Никакого обязательного демона/сервера.
- Детерминированные команды; машинный **JSON по умолчанию** + человеческие Markdown/CSV.
- Стабильные **exit-коды**, структурные ошибки, `--dry-run`, аудит.
- **Локально, без бэкенда.** Ядро-логика переиспользуема CLI / (опц.) демоном / сервером / UI.
- Демон/сервер — **опционально и позже**, без уникальной логики (те же сервисы, что CLI).

### 14.2. Отличие от replan — стек сцеплен с parse-бэкендом
У replan домен (планирование) нейтрален к языку → Go-first чист. У codemap домен — **разбор кода**,
тянет экосистема:
- **Python + `ast`/griffe** — Python-семантика даром, быстрый baseline; но Python-дистрибуция
  (смягчается `uv`/`pipx`; бинарь позже через Nuitka/PyInstaller).
- **Go/Rust + tree-sitter** — **single static binary как у replan** + мультиязычность; но резолв
  импортов (§3.1) строишь сам per-language, без griffe.

→ «быстрый универсальный бинарник» для codemap = **по сути то же решение, что §10.8/§12**, не отдельное.

### 14.3. JSON-стор разводит «сборку» и «отдачу» (смягчает вопрос бинаря)
Канонический JSON (§2.2) делит рантайм надвое:
- **Build** (Extract→Build→Store) — редкий, тяжёлый по языку. Python / Go-Rust / микс за адаптерами (§13).
- **Serve** (query/report/export/RAG над JSON) — **горячий, зовётся ИИ**. Просто читает JSON → быстрый
  **независимо от языка сборки**.

Отсюда: бинарь важнее всего для **serve-пути**; стор позволяет serve быть быстрым, даже если build на Python.

### 14.4. Для AI-hot: тёплый процесс > холодный бинарь
«ИИ зовёт много раз» решается не столько компиляцией, сколько **тёплым резидентным процессом**:
опциональный **MCP-сервер/демон** с графом в памяти → ноль startup на вызов, нативные tool-calls для
агента, любой язык. Как у replan — демон опционален и позже; но для AI-first это естественный serve-
интерфейс. Cold single-shot CLI — для людей/скриптов и MVP.

### 14.5. Рекомендация
1. **Сейчас — принять рантайм-хартию 14.1** (стек-независимо: JSON I/O, детерминизм, exit-коды, dry-run).
2. **Не делать бинарь первым.** Он не самоцель, а **следствие §10.8**: Go/Rust+tree-sitter → бинарь;
   Python+griffe → `uv`/`pipx`, бинарь потом.
3. **AI-скорость брать из тёплого MCP/демона** над JSON-стором, а не из компиляции всего.

> Итог: вопрос бинаря сворачивается в §10.8. Решишь parse-бэкенд — стек и форма дистрибуции определятся.

## 15. Свежесть графа: догоняние изменений кода

Код-граф — **производный артефакт из исходников** (по сути индекс/кэш кода). «Догонять изменения» —
классическая **инвалидация производных данных / инкрементальная индексация**, давно решённая. Не
изобретаем — подсматриваем.

### 15.1. Prior art (у кого решено)
| Класс | Примеры | Что берём |
|---|---|---|
| Build-системы | Bazel, Buck, Ninja | контент-хеш + граф зависимостей → пересобрать только затронутое |
| Инкрементальные компиляторы | Salsa (rust-analyzer), `tsc --watch` | demand-driven мемоизация: пересчитать лишь зависящее от изменения |
| Code-graph/index системы | Kythe, Glean, Sourcegraph | **прямые аналоги codemap** — схема + инкрементальный индекс на масштабе |
| File-watchers | Watchman, `watchdog` | эффективно узнать «что изменилось» без наивного полла |
| VCS/CI | git-хуки, CI-on-push | триггеры и дельта изменений |

### 15.2. Триггеры (слои, не «или»)
- **On-demand** (`codemap build`) — просто; между прогонами устаревает.
- **git-хуки** (post-commit/merge) — синк с коммитами; опционально, per-clone.
- **CI-on-push/PR** — авторитетная свежесть канона; медленнее.
- **Watcher-демон** (watch-mode) — для тёплого serve-режима (§14.4): резидент следит за ФС, обновляет
  инкрементально → самый свежий. Модель LSP/rust-analyzer.
- (LSP-события из редактора — позже, вероятно вне codemap.)

### 15.3. Гранулярность — и главная ловушка велосипеда
- **Полный ребилд** — распарсить всё заново. На bquant (63 модуля) это **<1с** → в v1 **правильный
  ответ**: просто, детерминированно, ноль багов инвалидации.
- **Инкремент (по файлу, контент-хеш)** — пересобрать только изменённые + их зависимых. Нужен на
  масштабе (тысячи файлов) или для sub-second watch-mode.
  **Ловушка:** кросс-файловая инвалидация нетривиальна — правка `__all__` в модуле A ломает резолв у
  всех, кто `import *` из A. Ровно это делают Salsa/Bazel — **не пиши сам, пока не приспичит.**

### 15.4. Коммитим граф или генерим (связано с §10.9)
- **Генерим (`.gitignore`)** — всегда свежий при `build`; не бывает устаревшим (не хранится). Но не
  ревьюится; для запросов нужен ребилд/демон.
- **Коммитим** — диффается в PR (виден дрейф графа), но может устареть → нужен **CI freshness-check**:
  «закоммиченный граф == свежий ребилд» (**переиспользуем паттерн `test_docs_parity`** — та же идея).

### 15.5. Что имеет смысл / что нет
- ✅ **v1:** полный ребилд on-demand + опц. git-хук + (если коммитим) CI freshness-check.
- ✅ **Позже:** контент-хеш-инкремент + watcher-демон — **заимствуя** Salsa/build-систему, не изобретая.
- ❌ **Не сейчас:** свой инкрементальный движок и LSP-интеграция для 63-модульного пакета — это и есть
  велосипед. Инкремент — самая частая ловушка переизобретения.

## 16. За пределами кода: система как граф (north-star)

Ядро codemap — **граф типизированных сущностей + типизированных рёбер с подключаемыми экстракторами**.
Код-символы — **домен №1**, но модель от кода не зависит и естественно обобщается в **граф знаний о
системе** (микросервисы, серверы, деплой, инфра). Это подтверждает архитектурные решения (нейтральное
ядро, адаптеры, канон-vs-индекс были правильны именно потому, что обобщаются).

**Как обобщается:**
| | Код (домен №1) | Система/инфра (будущие домены) |
|---|---|---|
| Узлы | module / class / function | service / server / container / queue / DB / endpoint / deploy-unit |
| Рёбра | imports / inherits / calls | calls-over-http / deploys-to / reads-from / publishes-to / depends-on |
| Экстракторы | griffe / go-packages | IaC (Terraform/k8s/compose) · API (OpenAPI) · рантайм (OTel-трейсы / интроспекция сервера) |

**Prior art (не изобретаем):** Backstage (каталог сервисов), cartography (граф AWS-инфры), Terraform
graph, OpenTelemetry service maps, Kubernetes topology. codemap — их обобщение на одном граф-ядре.

**Что это меняет в дизайне сейчас (дёшево, двери открытыми):**
- **Открытые kind** (§2) — новые сущности/рёбра без хирургии модели.
- **Сменный query-бэкенд** (§4) — на system-scale (граф микросервисов+инфры, тяжёлый traversal)
  оправдан **Neo4j**; переключение = новый query-адаптер, ядро/экстракторы не трогаются.
- **Домен-агностичное имя** — «код-граф» это домен №1 «графа знаний о системе».

**Дисциплина:** v1 = код-граф **одного Python-пакета**. Система/инфра — north-star; двери держим
