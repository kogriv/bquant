## 2. Ось развития (зафиксирована 2026-07-24)

Цель — **headless-ядро (AI-first), на которое навешиваются интерфейсы.** Ядро — для
реального потребителя *сейчас* (владелец + ИИ-агенты), а не для внешней человеческой
аудитории, которой ещё нет. Человеческий вид — последним.

Ключевое решение: **начинаем не с доки и не с CLI, а с графа кода** — потому что дока
описывает код, а CLI вызывает код; без разобранной карты кода и то, и другое делается
вслепую (в уборке мы не раз ловили доку, описывающую несуществующий API).

```
[P1: Граф кода]  ← фундамент headless-ядра (факты о пакете)   ← начинаем ЗДЕСЬ
      │
      ├─→ [P2-B: Дока-как-продукт]   (для людей и ИИ / RAG)
      └─→ [P2-A: CLI-AI тул]          (ИИ гоняет анализ, структурный I/O)
                 │
            [P3: Human-view]  ← сайт / дашборды / визуализация — в самом конце
```

**Граф — это данные внутри ядра**, а не отдельная третья вещь. Дока (B) и CLI-тул (A) —
два клиента на одном фундаменте. Цель «полноценно = A+B», но у обоих общий корень — граф.

## 3. Стадии

### P1 — Граф кода (фундамент ядра) ← текущий фокус

**Инструмент: `codemap/`** (рабочее имя, решено 2026-07-24). Отдельный dev-тул общего
назначения — «ест пакет, выдаёт граф». Ключевые архитектурные решения:
- **Свой тулинг** (не готовые инструменты), на `ast`/import-graph.
- **Изолирован в корне репо, вне `bquant/`** → НЕ попадает в wheel (pyproject пакует только
  `bquant`). Прицел — чистый вынос в отдельный репо-продукт позже (`git mv codemap/`).
- **BQuant для него — вход, а не зависимость:** статический разбор *исходников*, без импорта
  рантайма. Поэтому работает над любым пакетом и не сцеплен с внутренностями bquant.
- Своя мини-структура: `codemap/README.md`, код, `codemap/tests/`.

```
/ (repo)
├── bquant/     ← подопытный пакет = ВХОД для codemap
├── codemap/    ← изолированный тул (будущий отдельный репо)
├── docs/  tests/  ...
```

Факт-карта пакета, построенная **под конкретное применение** (каждый кусок отвечает на
реальный вопрос доки или CLI). Кандидатное наполнение:
- **Структура:** модули/пакеты, публичная API-поверхность (что экспортируется, точки входа).
- **Связи:** рёбра импортов/зависимостей (что от чего зависит).
- **Живое/мёртвое:** недостижимый код, затенённые/дублирующие имена (прежние кандидаты
  `indicators/library.py`-тень и `base_old.py` из старого плана уже вычищены — capability остаётся).
- **Привязка к докам:** докстринги/примеры как данные — вход для генерации/парити доки.
- **Выход:** структурный артефакт (граф как данные), который потребляют и P2-B, и P2-A.

**Не с нуля:** уборка уже дала кусок правды — `test_docs_parity.py`, актуальный `AGENTS.md`,
разбор D1. Граф это формализует и расширяет.

**Про конечность (уточнено):** базовая карта — **структура + модель данных + зависимости +
API-поверхность — конечна и завершаема** (статический `ast`-разбор детерминирован). Бесконечны
не она, а *опциональные слои глубины*: точная трассировка потоков данных через рантайм,
смысловые/NL-описания узлов, эмбеддинги, обработка любого динамического питон-трюка. Их
добавлять только когда конкретный потребитель (дока/CLI) попросит — не авансом.

### P2 — Интерфейсы поверх графа (A + B)
Два клиента на общем фундаменте P1. Порядок между A и B — решение §4.
- **P2-B — Дока-как-продукт:** единый источник → рендеры; дока генерится/валидируется
  против графа (ground truth); AI-friendly экспорт (RAG питается графом+доками). Это
  «зрелая» версия того, что уборка подготовила.
- **P2-A — CLI-AI тул:** аналитическая функциональность экспонирована чисто и
  AI-драйвабельно (структурный вход/выход, возможно MCP). Зреет из существующего слабого
  `bquant/cli.py` (сейчас демо на deprecated-пути), опираясь на API-поверхность из графа.

### P3 — Human-view (в конце)
Рендеренный вид для людей: сайт доков / дашборды / визуализация зон. Последним — потому
что внешней человеческой аудитории пока нет (в хартии поэтому же отложен английский).
