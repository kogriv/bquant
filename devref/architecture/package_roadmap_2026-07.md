## 3. Предлагаемые стадии (к обсуждению)

Порядок — от «есть фундамент, дешёвый следующий шаг» к «greenfield».

### P1 — Доки-как-продукт: единый источник → рендеры
Самое подготовленное направление. Кандидатные шаги:
- **P1.1** Автосборка сайта из текущих доков (Sphinx → GitHub Pages / RTD), CI-хук.
  У нас build уже проходит локально — остаётся автоматизировать и опубликовать.
- **P1.2** AI-friendly экспорт: один структурированный бандл доков (плоский md/JSON) —
  вход для RAG и для ИИ-агентов. Дёшево, ложится на parity-инфраструктуру.
- **P1.3** Зафиксировать «единый источник» как процесс: что канон, откуда генерятся виды,
  как parity-тест это стережёт.

### P2 — Граф кода + RAG-на-минималах (greenfield)
- **P2.1** Граф связей кода (модули/импорты/вызовы) — карта пакета для навигации.
- **P2.2** «Минималы» + RAG-поиск поверх кода и доков.
- Зависит от P1.2 (AI-friendly экспорт — часть входа).

### P3 — Качество / DX (точечно, низкий приоритет)
- Косметические doc-warnings (ссылки на директории/repo-файлы, header-level, highlighting).
- Финализировать «тесты одной командой».
- v3.0.0 (выпил legacy `MACDZoneAnalyzer`) — отдельная веха, когда созреет.
