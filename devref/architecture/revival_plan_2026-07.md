# BQuant — План реанимации проекта (2026-07)

**Дата:** 2026-07-06
**Тип:** Стратегическое направление (take + roadmap)
**Статус:** 🟡 Черновик к обсуждению
**Контекст:** Проект активно пилился ~год назад (последние фазы — октябрь-ноябрь 2025),
затем пауза. Документ фиксирует оценку текущего состояния и предлагает порядок
действий для возобновления работы.

---

## 0. TL;DR

- Проект **не «колхоз» по архитектуре** — скорее переусложнён для v0.0.1. Каркас
  (Strategy + Registry + DI + Protocol + fluent Builder, 682 теста) выше среднего.
- Он **«велосипед» в деталях**: индикаторы переписаны вручную при наличии
  `pandas-ta` в обязательных зависимостях, причём собственная реализация даёт
  **другие числа** (EMA через `ewm(span=...)` без `adjust=False`).
- Главная цель реанимации — **вернуть петлю обратной связи**, а не большой
  рефакторинг. Рефакторинг без внешнего результата усыпляет проект повторно.
- **Порядок:** сначала «страшный вопрос» (числам можно верить? тулкит что-то
  показывает?), потом публикация результата, и только под нужды этого — уборка.
  Двуязычность документации и разбивку репо — отложить.

## 1. Оценка текущего состояния («take»)

### 1.1. Где это НЕ колхоз (сильные стороны)

- **Настоящие паттерны, не для галочки:** Strategy + два независимых декораторных
  Registry (детекция зон `ZoneDetectionRegistry` и признаки `StrategyRegistry`),
  Dependency Injection в `UniversalZoneAnalyzer`, fluent Builder
  (`analyze_zones().with_indicator().detect_zones().analyze().build()`),
  `@runtime_checkable` Protocol'ы + стандартизованные dataclass-метрики.
- **Динамическая интеграция pandas-ta** через интроспекцию сигнатур и генерацию
  классов `type()` — осознанно и нетривиально.
- **Дисциплина тестов:** ~682 теста, ~63% покрытия, 100% pass; `zodoctest/` —
  валидаторы примеров документации против кода (редкость для пет-проектов).
- **Единый источник истины** (`bquant.analysis`), deprecation вместо ломающих
  изменений, кэш-ключи для воспроизводимости.

### 1.2. Где это «велосипед» (переизобретение)

- **Индикаторы.** `pandas-ta` — обязательная зависимость, но MACD/RSI/EMA/SMA/
  Bollinger написаны руками. MACD физически присутствует в ~5 местах:
  `custom/macd.py`, `library.py`, `preloaded/macd.py`, `library/talib.py`,
  обёртки в `calculators.py`. «Кастомный» MACD — 12 строк расчёта в 179 строках
  boilerplate.
- **Инфраструктура.** Свой двухуровневый кэш (`core/cache.py`, 534 стр.) вместо
  `joblib`/`diskcache`; свой цветной логгер (`core/logging_config.py`, 748 стр.)
  вместо `loguru`/`rich`; свой notebook-раннер (`core/nb.py` `NotebookSimulator`)
  вместо Jupyter/`papermill`/`jupytext`; свой профайлер (`core/performance.py`,
  554 стр.) поверх `psutil`. Каждое по отдельности оправдываемо, в сумме —
  «своя мини-платформа».
- **Данные как код.** `samples/embedded/tv_xauusd_1h.py` — ~17 000 строк
  Python-литералов вместо parquet/csv в `package_data`.

> ⚠️ **Важная оговорка (позиция владельца, принята):** трёхисточниковая модель
> индикаторов (внешние библиотеки / внутренняя реализация / preloaded готовые
> числа → одинаковая форма результата) — это **осознанный и хороший дизайн**.
> Критика направлена не на неё, а на то, что внутренняя реализация одновременно
> (а) буквально дублируется (`library.py` vs `custom/`) и (б) даёт числа,
> расходящиеся с библиотечными. Правильный фикс — не «убрать абстракцию», а
> «свести к одной доверенной реализации и покрыть тестом эквивалентности».

### 1.3. Где «колхозные» запахи (гигиена)

- **Тонкая некорректность математики.** Везде `ewm(span=...).mean()` **без
  `adjust=False`** (`custom/macd.py:81-82`, `library.py:308`,
  `calculators.py:269`). Стандартный технический EMA (TA-Lib, pandas-ta) —
  рекурсивный (`adjust=False`). Итог: «custom MACD» ≠ «talib MACD» ≠ pandas-ta.
  Для проекта, чья суть — *статистика по этим числам*, это не косметика.
- **Мёртвый/дублирующийся код:** `indicators/base_old.py` (24 КБ legacy),
  `indicators/library.py` (17 КБ, параллельные копии индикаторов из `custom/`).
- **Три идентичных файла-гайда:** `CLAUDE.md` = `GEMINI.md` = `VIBE.md`
  (побайтно, один md5) — гарантированно разъедутся.
- **`numpy_fix.py`** — monkeypatch глобального numpy при импорте
  (`np.NaN = np.nan`) ради совместимости со старой pandas-ta.
- **Гипертрофия процесса:** `devref/` ≈ 81 markdown / 22 580 строк против 91 файла
  кода; десятки `phaseX_completion_report.md` / `_final_summary.md`. Признак
  AI-ассистированной разработки: церемонии больше, чем продукта. (Владелец
  согласен — избыток; связано с интенсивной разработкой год назад.)

### 1.4. Вердикт

| Ось | Оценка |
|-----|--------|
| Архитектура каркаса | Не колхоз; скорее over-engineering для v0.0.1 |
| Реализация индикаторов | Велосипед + расхождение в числах |
| Инфраструктура (кэш/логи/nb) | Умеренный велосипед, но аккуратный |
| Гигиена репозитория | Колхозновато: дубли, `*_old`, 3 копии гайда, 22k строк доков |
| Корректность математики | Слабое место: `ewm` без `adjust=False` |

**Одной фразой:** переусложнённый, добросовестно оттестированный каркас, который
зачем-то заново изобретает индикаторы (местами неправильно) и обвешан
AI-сгенерированной документацией процесса. Главный риск — не «стиль», а то, что
разные реализации MACD/RSI дают чуть разные числа, а весь смысл проекта —
статистика по этим числам.

## 2. Стратегический рефрейм

Из пяти рассматриваемых направлений только **одно является целью**; остальные —
каналы для неё или обслуживание. Проект стоял год. Оживляет его **петля обратной
связи** (рисёрч → публикация → отклик), а не рефакторинг. Поэтому пять «путей» —
это не меню на выбор, а граф зависимостей.

| # | Путь | Что это на самом деле | Когда |
|---|------|----------------------|-------|
| 1 | Поиск альфы / практика | **Единственная цель.** Проверяет, стоит ли пакет уборки | **Сейчас, спина всего** |
| 5 | Блог/трансляция рисёрчей | Выходной канал для #1; forcing function «опубликовал = сделал» | Сразу за #1, в лёгкой форме |
| 4 | AI-CLI-friendly тулинг | Множитель для соло-разработки, дешёвый | Инкрементально, попутно |
| 3 | «Сделать красиво» (код + доки) | Обслуживание; нужное, но не самоцель. Англ — потом | Точечно под нужды #1 |
| 2 | Разделить репо | Преждевременно для соло без внешних юзеров | Отложить |

**Принцип последовательности:** начинать с самого *страшного* вопроса
(«числам можно верить и тулкит что-то показывает?»), а не с самого *приятного*
(«переберём структуру и переведём доки»). Первое оживляет, второе усыпляет.

## 3. Разбор по направлениям

### 3.1. #1 — Поиск альфы / практическое применение (ЦЕЛЬ)

**Обязательный шлюз перед стартом — корректность чисел.** Пока «свой MACD» ≠
«talib MACD» ≠ pandas-ta, любой статвывод стоит на песке. Это НЕ большой
рефакторинг, а хирургия на 1–2 дня:

- выбрать **одну** доверенную реализацию (кандидат — pandas-ta под капотом, раз он
  и так в зависимостях; либо явный recursive-EMA с `adjust=False`);
- `custom/*` либо делегирует в неё, либо покрывается тестом «совпадает с эталоном
  до N знаков»;
- убить буквальный дубль `library.py`.

Это и есть ответ на «велосипед»-вопрос, продиктованный не эстетикой, а тем, что
#1 без него бессмысленен. Остальные велосипеды (кэш/логи/nb) на альфу не влияют —
не трогаем.

**Честно про альфу:** на MACD-зонах золота дюрабельную альфу найти шансов мало
(индикатор лагающий и заезженный). Но ценность проекта — не «денежный принтер», а
**методологический каркас**: честный, воспроизводимый zone-analysis для *любого*
индикатора. Цель переформулируется как «доказать, что тулкит даёт статистически
честные, воспроизводимые результаты — включая честные *отрицательные*». Это
защищает мотивацию от вероятного нулевого результата и даёт контент для #5:
строгий разбор с null-результатом — полноценный рисёрч.

**Сырьё уже есть:** поздние ноутбуки (`research/notebooks/05_case_study_*`,
`06_swing_strategy_comparison`), `docs/analytics/zones/*case_study.md`, планы в
`devref/architecture/future_implementation_plan.md`.

### 3.2. #5 — Блог/трансляция рисёрчей (КАНАЛ)

Лучший forcing function для оживления: обязательство публиковать создаёт ритм.
Инфраструктуру раньше контента не строить.

- **Quarto / статика из ноутбуков** — если материал «прочитал → понял» (репорты,
  графики, выводы). Дёшево, версионируется, GitHub Pages. **Рекомендуемый старт.**
- **Streamlit** — если нужна интерактивность (покрутить параметры детекции,
  посмотреть зоны вживую). Тяжелее в поддержке; добавить, когда упрёшься в статику.
- **Dash** — избыточен для соло-блога; только если Streamlit не хватит.

Готовое сырьё: `NotebookSimulator`-скрипты и рендеры в `notebooks/outputs/`.

### 3.3. #4 — AI-CLI-friendly тулинг (МНОЖИТЕЛЬ)

Дешёвый множитель для соло+ИИ, делать попутно, не фазой:

- свести три идентичных гайда (`CLAUDE/GEMINI/VIBE.md`) к одному источнику
  (генерация/symlink);
- рабочий session-start hook (в репо уже есть `chore(cloud)` коммит про hook);
- «тесты одной командой»; актуальный `CLAUDE.md`, отражающий реальность.

### 3.4. #3 — «Сделать красиво» (ОБСЛУЖИВАНИЕ)

Правильно по духу («doc-driven, доки отражают реальность» — инфраструктура под это
есть: `zodoctest/`). Держать доки честными — да.

**Но не гнать двуязычность сейчас.** Англ-primary + полный синк с русской +
перевод всех комментариев — огромный налог с нулевым текущим спросом (аудитория =
один человек). Правило: язык primary определяется аудиторией, а её пока нет.
Отложить до появления внешнего читателя.

### 3.5. #2 — Разделить репо (ОТЛОЖИТЬ)

Разбивка на 3 репо (доки / ядро / рисёрчи) для одного человека без потребителей =
кратный рост боли (версионирование, CI ×3, кросс-репо изменения). Монорепо —
правильный выбор. Дёшево и полезно сейчас — **не репо, а каталоги**: заархивировать
процесс-доки (`devref/` → `devref/archive/`), чётче отделить «пакет» от
«песочницы» директориями. Реальный сплит — когда у пакета появятся внешние юзеры и
захочется публиковать на PyPI отдельно от рисёрчей.

## 4. Конкретный план — первые ~2 недели

### Неделя 1

1. **Шлюз корректности (1–2 дня).**
   - Замерить расхождение custom-MACD/RSI vs pandas-ta на сэмпле (`tv_xauusd_1h`).
   - Свести к одной доверенной реализации (делегирование или `adjust=False`).
   - Тест эквивалентности «совпадает с эталоном до N знаков».
   - Выпилить `library.py`, `base_old.py` (проверив по тестам безопасность).
2. **Один сквозной рисёрч (остаток недели).**
   - Взять гипотезу из поздних ноутбуков.
   - Прогнать `analyze_zones()` от данных до статвыводов.
   - Оформить как отчёт. Цель — не альфа, а «пайплайн работает end-to-end и я ему
     верю».

### Неделя 2

3. **Опубликовать.**
   - Quarto-страница с разбором на GitHub Pages. Первый пост = первый пульс.
4. **Попутная гигиена.**
   - Свести AI-гайды к одному источнику.
   - «Тесты одной командой», актуализировать `CLAUDE.md`.

### Дальше — ритм

`рисёрч → публикация → мелкая чистка под нужды следующего рисёрча`.
Уборку и англ-версию подтягивать **под растущую аудиторию**, а не авансом.
